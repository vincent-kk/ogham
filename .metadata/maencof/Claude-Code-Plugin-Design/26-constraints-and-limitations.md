# 한계와 제약 — 규모/의미/실시간 한계 + Pre-mortem

## 1. 규모 한계

| 규모 | 예상 성능 | 비고 |
|------|----------|------|
| ~1,000 문서 | 원활 | 빌드 <5초, 쿼리 <50ms |
| ~5,000 문서 | 양호 | 빌드 <30초, 쿼리 <100ms |
| ~10,000 문서 | 임계점 | JSON 파싱 지연, 메모리 수백 MB |
| 10,000+ | 한계 초과 | SQLite 마이그레이션 필요 |

관련 문서: [메타데이터 전략](./12-metadata-strategy.md) | [설계 결정](./25-design-decisions.md)

---

## 2. 의미론적 한계 (Phase 2까지)

- 자연어 이해 불가 (키워드+그래프 의존)
- 동의어/유의어 인식 불가
- 명시적 링크/디렉토리 외 관계 추론 불가

---

## 3. 실시간성 한계

- PageRank, 커뮤니티: 전체 빌드 시에만 갱신
- 대규모 구조 변경: 전체 재인덱싱 필요
- 인덱스 신선도: stale 표시만 가능, 자동 재인덱싱 아님

---

## 4. 이 도구가 하지 않는 것

1. 원본 마크다운 자동 수정 (읽기 전용)
2. LLM API 키 관리 (환경 변수에 위임)
3. 비마크다운 파일 내용 분석
4. 인덱스를 진실의 원천으로 사용 (파생물 원칙)
5. 과도한 컨텍스트 주입 (토큰 예산 2000 제한)

---

## 5. Pre-mortem 실패 시나리오

### F1: "완전한 구현" 시도로 출시 불가
P0~P3 모두 구현 시도 → 외부 의존성에 막혀 반제품.
**완화**: Phase 구분 엄격 준수. P0만으로 MVP.

### F2: 메타데이터 동기화 실패로 신뢰 상실
삭제/이동 문서가 검색에 포함 → 사용자 불신.
**완화**: 쿼리 시 파일 존재 검증, 신선도 표시, stale 10% 경고.

### F3: AI/인간 이중 인터페이스로 자원 분산
양쪽 중도 완성 상태.
**완화**: AI 우선 설계, 인간은 Skill 포맷팅 레이어만.
