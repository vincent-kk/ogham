## **8\. Layer 5 재정의: Buffer/Boundary 이중 역할 모델**

### **8.1. 기존 Layer 5의 역할과 한계**

아키텍처 버전 1.0.0에서 Layer 5(맥락 메타데이터)는 인물(`persons/`)과 도메인(`domains/`) 정보를 분류하는 단일 목적의 보조 메타데이터 계층으로 설계되었다. 다른 레이어의 문서들이 참조하는 '메타데이터 노드'로서, 직접적인 지식을 담지 않고 지식의 맥락(누구와 관련된 지식인가, 어떤 도메인인가)을 제공하는 보조 역할을 수행했다.

그러나 Layer 3의 3차원 방향성 분할([§7](./07-l3-directional-expansion.md) 참조)과 함께 Layer 5의 역할 재정의가 불가피해졌다. 두 가지 구조적 공백이 확인되었다:

1. **미분류 정보의 부재**: 아직 어떤 레이어에도 속하지 않는 원시 정보(대화 조각, 임시 메모, 농담, 단편적 스크랩)를 수용할 공간이 없었다. 이러한 정보는 Layer 3이나 Layer 4에 억지로 배치되거나 아예 기록되지 못했다.
2. **교차 연결 메커니즘의 부재**: L3A/3B/3C로 분할된 외부 지식 간의 교차 연결을 수행할 구조적 허브가 없었다. 분할이 지식의 사일로화(Siloing)로 귀결되는 것을 방지하려면, 서로 다른 방향성의 지식을 특정 실행 맥락 안에서 통합하는 메커니즘이 필수적이다.

### **8.2. L5-Buffer: 미분류 정보의 임시 저장소**

L5-Buffer는 아직 범주화되지 않은 정보의 임시 수용 공간이다. 지식 관리 생태계에서 모든 정보가 생성 즉시 적절한 레이어로 분류되는 것은 비현실적이다. 일상적인 대화에서 건진 단서, 우연히 마주친 아이디어, 맥락이 불명확한 메모 등은 즉각적인 분류가 불가능하거나 불필요한 경우가 많다.

* **트리 구조:** `/05_Context/buffer/` 디렉토리
  * 시간순 또는 키워드 기반 명명 (예: `buf-snippet-2026-03-04.md`, `buf-chat-quantum-hint.md`)
* **문서 스키마 (Frontmatter):**
  ```yaml
  layer: 5
  sub_layer: buffer
  buffer_type: snippet | conversation | unclassified
  promotion_target: 3A | 3B | 3C | 2     # 시스템이 제안하는 승격 대상
  source_context: "대화 중 멘션" | "웹 스크랩" | "세션 메모"
  expires: 2026-04-03                      # 기본 30일 후 만료
  ```
* **그래프 특성:** **Temporary 노드**. 다른 레이어와의 연결이 미확정 상태이다. 다른 레이어의 문서가 Buffer 노드를 참조하는 것은 금지된다(참조 불안정성 방지). Buffer 노드 자체는 잠정적 아웃바운드 링크를 가질 수 있다.
* **SA 감쇠 인자:** 0.95 (극히 높음 — 미분류 정보는 최소 범위만 확산)

#### **Buffer 정보의 수용 범위**

L5-Buffer에 들어가는 정보 유형:
* 아직 범주화되지 않은 대화 조각 및 임시 메모
* 맥락이 불명확한 웹 스크랩 및 스니펫
* 농담, 일화, 인상적인 표현 등 분류 기준이 모호한 정보
* 세션 종료 시 보존 결정이 내려졌으나 적절한 레이어가 즉시 결정되지 않은 L4 노드

#### **승격(Promotion) 규칙**

L5-Buffer의 정보는 다음 조건에서 다른 레이어로 승격된다:

| 승격 경로 | 조건 | 트리거 | Level 0-1 | Level 2-3 |
| :---- | :---- | :---- | :---- | :---- |
| L5-Buffer → L3A | 인물 관련 정보로 분류됨 | 사용자 분류 또는 시스템 제안 | 명시적 승인 | 제안 후 확인 |
| L5-Buffer → L3B | 조직/집단 관련 정보로 분류됨 | 사용자 분류 또는 시스템 제안 | 명시적 승인 | 제안 후 확인 |
| L5-Buffer → L3C | 주제/개념 관련 정보로 분류됨 | 사용자 분류 또는 시스템 제안 | 명시적 승인 | 자동 전이 |
| L5-Buffer → L2 | 이미 처리된 지식의 직접 내재화 | 사용자 판단 | 명시적 승인 | 자동 전이 |
| L5-Buffer → 삭제 | `expires` 경과 AND `accessed_count` = 0 | SessionStart 정리 | 명시적 승인 | 제안 후 확인 |

시스템의 자동 분류 제안은 다음 휴리스틱에 기반한다:
* Frontmatter의 `tags`에 인물명 포함 → L3A 제안
* 조직/팀/커뮤니티 키워드 탐지 → L3B 제안
* 기술/학문/개념 키워드 탐지 → L3C 제안 (기본값)
* AI 기반 내용 분석으로 분류 정밀도 향상 (Phase 3, 선택적)

### **8.3. L5-Boundary: 경계 객체(Boundary Objects) 계층**

L5-Boundary는 수잔 스타(Susan Leigh Star)가 창안한 '경계 객체(Boundary Objects)' 이론⁵⁸에 기반하여, L3A/3B/3C 간의 교차 연결을 수행하는 구조적 허브 계층이다.

경계 객체란, 서로 다른 지식 체계나 집단적 경계(구문론적, 의미론적, 화용론적 경계)를 가로질러 의사소통과 지식 공유, 의미 생성을 가능하게 해주는 아티팩트(Artifact)를 일컫는다.⁵⁸ 혁신 프로젝트에서 프로토타입, CAD 도면, 공유 데이터베이스, 대시보드 등이 대표적인 경계 객체로 기능한다.⁵⁸

PKM 환경에서 경계 객체의 역할은 **'메타데이터를 통한 구조적 링킹 노트'** 또는 **'프로젝트 MOC(Map of Content)'**로 발현된다.

* **트리 구조:** `/05_Context/boundary/` 디렉토리
  * 경계 객체별 파일을 배치 (예: `bnd-project-alpha-dashboard.md`, `bnd-career-roadmap.md`)
* **문서 스키마 (Frontmatter):**
  ```yaml
  layer: 5
  sub_layer: boundary
  boundary_type: syntactic | semantic | pragmatic
  connected_layers: [3A, 3B, 3C]    # 이 경계 객체가 연결하는 서브레이어
  purpose: "프로젝트 Alpha의 인물·조직·기술 통합 대시보드"
  ```
* **그래프 특성:** **Bridge 노드 (높은 fan-out)**. `CROSS_LAYER` 엣지 타입을 통해 서로 다른 방향성의 외부 지식을 특정 실행 맥락 안에서 통합한다. 모든 L3 서브레이어의 문서들로부터 인바운드 참조를 받으며, 동시에 해당 문서들에 대한 아웃바운드 링크를 생성한다.
* **SA 감쇠 인자:** 0.60 (낮음 — 경계 객체는 광범위 확산하여 레이어 간 다리 역할 수행)

#### **경계 유형과 통합 메커니즘**

경계 객체는 세 가지 지식 간 경계 유형에 대응한다:⁶⁰

| 경계 유형 | L3 서브레이어 간 갈등 요소 | 경계 객체 적용 예시 |
| :---- | :---- | :---- |
| **구문론적 경계** (Syntactic) | L3A(전문가의 구두 조언)와 L3C(텍스트 기반 학술 논문) 간의 형식적 차이 | 조언 내용과 문헌의 핵심을 공통 태그 및 용어집으로 통합 정리한 요약 노트 |
| **의미론적 경계** (Semantic) | L3B(조직의 관행적 언어)와 L3C(보편적 학술 언어) 간의 해석 차이 | 조직 내에서 해당 학술 개념이 어떻게 쓰이는지를 재정의한 매핑 다이어그램 |
| **화용론적 경계** (Pragmatic) | L3A(개인의 이해관계)와 L3B(조직의 목표) 간의 실천적 이해 충돌 | 개인의 성과 목표와 조직의 KPI를 동시에 충족시키기 위한 프로젝트 기획서(MOC) |

행위자-네트워크 이론(ANT)의 언어로, 분할된 각 행위자(사람, 집단, 개념)들은 PKM 시스템 내부에서 이 경계 객체를 '의무 통과점(Obligatory Passage Point, OPP)'으로 삼아 상호작용하고, 서로를 변형(Translation)시키며 새로운 지식 네트워크의 질서를 확립하게 된다.⁴⁹

### **8.4. 기존 persons/domains 디렉토리와의 관계**

기존의 `05_Context/persons/`와 `05_Context/domains/` 디렉토리는 그대로 유지된다. 이들은 L5-Boundary의 특수한 인스턴스로 간주할 수 있다:

* `persons/`: 인물 메타데이터 노드. L3A(관계적 외부)의 문서들이 참조하는 인물 프로파일의 경량 메타데이터.
* `domains/`: 도메인 메타데이터 노드. L3C(주제적 외부)의 문서들이 참조하는 도메인 분류 체계.

이들은 특정 서브레이어와 1:1로 대응하는 메타데이터이므로, 다중 서브레이어를 교차 연결하는 Boundary 객체와는 구분된다. 그러나 `persons/`의 인물이 여러 조직(L3B)과 여러 주제(L3C)에 걸쳐 활동하는 경우, 해당 인물 노드는 사실상 Boundary 역할을 수행하게 된다.

### **8.5. 다른 레이어와의 상호작용 패턴**

| 상호작용 | 방향 | 설명 |
| :---- | :---- | :---- |
| L5-Buffer → L3A/B/C | 승격 | 미분류 정보가 분류되어 적절한 서브레이어로 이동 |
| L5-Buffer → L2 | 직접 내재화 | 이미 처리된 지식이 바로 파생된 자아로 승격 (드문 경우) |
| L5-Buffer → 삭제 | 만료 | 설정된 기간 경과 후 미참조 정보 삭제 |
| L5-Boundary ↔ L3A/B/C | 양방향 교차 연결 | 경계 객체가 서브레이어 간 다리 역할 수행 |
| L5-Boundary → L4 | 행동 맥락 주입 | 경계 객체가 행동/작업 기억에 통합 맥락 제공 |
| L5-Boundary ← L4 | 프로젝트 생성 | 행동 맥락에서 새로운 경계 객체(프로젝트 MOC) 생성 |
| 모든 레이어 → L5-Buffer | 금지 | 임시 저장소 참조는 불안정성 유발 |

이러한 이중 역할 재정의를 통해, Layer 5는 단순한 메타데이터 저장소에서 지식 생태계의 **입수 관문(Buffer)**이자 **통합 허브(Boundary)**로 진화한다.
