## 개요

2026-03-28 세션에서 이슈 분해 파이프라인 설계를 v1(원본 설계문서)에서 v2로 발전시킨 논의 기록.
Vincent가 사전에 작성한 전체 설계 문서를 기반으로, 핵심 구조적 결정 7건을 도출.

---

## 결정 1: 문제 공간과 해법 공간의 분리

**논의 과정**: 기존 v1은 Phase 0~5 선형 파이프라인이었음. Vincent가 "최종 분할 이후 개발티켓으로 발전하는 지점은 재귀적 이슈분할과 다른 성질의 문제"라고 제기.

**Vincent의 근거**:
- 재귀 분할은 도메인 지식만으로 가능 (문제 공간)
- dev 티켓 생성은 코드베이스/아키텍처 이해 필수 (해법 공간)
- 문제공간의 최종분할 티켓을 병합해서 더 적은 수의 dev 티켓을 만들 수도 있음 — 코드 발산 억제 목적

**Fionn의 확장**:
- N:M 매핑 구조 제안. M < N (병합) 뿐 아니라 M > N (하나의 요구사항이 API+프론트+DB 등 여러 레이어에 걸치는 경우) 역방향도 존재
- 이 전환점에서 기존 `/plan-issue`의 Demo-at-Every-Step과 `@temp` 앵커가 합류

**결정**: 파이프라인을 Stage A(문제 공간) + 합류점 + Stage B(해법 공간)으로 명확히 분리

## 결정 2: 정형화 = 내부 유틸, 재귀 분할 = base+step 패턴

**Vincent의 제안**:
1. 비정형→정형 분리작업은 모든 입력에 필요 → "내부사용 스킬" 형태
2. 모든 작업은 재귀적 동작: base(종료조건) + step(분할규칙)
3. 리뷰 깊이는 이슈 트래커 메타데이터(Jira tag/component, GitHub label)로 제어

**Fionn의 보완**:
- 정형화: 내부 유틸이되, 출력은 투명해야 함 (인간이 확인 가능)
- 리뷰 깊이: 고정값이 아니라 조건부 — 정적 설정 + 동적 격상(E3-4 발생 시 해당 브랜치 전체 리뷰 전환)

**Vincent 확정**: 정형화는 충분한 완성도를 보장해서 되도록 사용자에게 투명한 동작이길 바람

---

## 결정 3: 아키텍처는 인간 영역 — 도구는 맥락 조립만

**논의 과정**: 합류점에서 아키텍처 문서가 필요하다는 인식 공유 후, "아키텍처가 없는 경우"와 "파괴적 변경이 필요한 경우"를 어떻게 처리할지 논의.

**Vincent의 판단**:
- 두 경우 모두 자동 파이프라인 불가 영역
- 아키텍처는 개발자에게 상위기획과 같은 성격 — 인간 리뷰 필수 영역
- 도구의 역할: 문제범위/문제정의/현재상황 등 **아키텍처링을 할 수 있는 기반정보 제공** (기술적인게 아니라 기획/스펙적으로)
- 티켓을 만들어서 인간 개발자에게 위임. 아키텍처링 자체는 별도 도구로

**핵심 인용**: "설계없는 개발이 얼마나 끔찍할지는 너도 잘 알고있잖아?"

**Vincent의 현재 실천** (line-lore `.metadata/` 사례):
1. 광범위 분석 → raw 분석 문서
2. 분석 기반 도구 선정
3. Blueprint 문서 생성 + 분석 문서 링크
4. Blueprint → Plan 매핑, 아키텍처 문서 분리
5. Plan + 아키텍처 기반 개발

## 결정 4: 두 트리 구조 (기획 + 기술) — C안 채택

**의사결정 과정**:
- B안 (기획 트리 + 합류, 기술은 외부 입력) 에서 출발
- 논의를 전개하며 C안 (기획 트리 + 기술 트리 + 합류 전체 커버)이 필요하다고 판단

**이유**: 기획 트리만으로는 불완전. 기술 파이프라인(T1~T4)의 독립 성장과 합류점 구조를 명시적으로 다뤄야 전체 그림이 완성됨.

---

## 결정 5: 크로스 피드백 불필요

**논의 과정**: 기획 트리 ↔ 기술 트리 사이의 양방향 피드백을 모델링할지 검토.

**Vincent의 판단과 근거**:
- 양 트리가 독립 성장하는 경우는 "최초 개발"뿐
- 이후에는 아키텍처가 고정되고, 필요에 의해 개선하는 구조
- 개발자가 원해서 아키텍처 리팩토링 → 스펙과 무관, 파이프라인 범위 밖
- 스펙을 위한 아키텍처 확장 → 독립적이 아니라 스펙에 종속적 (단방향)

**결정**: 크로스 피드백 제거. 모델이 상당히 단순해짐.

---

## 결정 6: Spec 티켓 포맷 — EARS + 메타데이터

**논의 과정**: Vincent가 EARS 같은 표준화된 스펙 정의 형식을 따르자고 제안. "오해없이 스팩을 기술할 수 있는 형식언어"로서.

**Fionn의 분석**:
- EARS는 요구사항 **문장**의 모호성을 제거하는 템플릿 — spec 본문으로 적합
- 하지만 합류점이 소비하려면 EARS 문장 외에 구조적 메타데이터 필요
- spec 티켓은 **순수하게 문제 공간만 기술** — 해법 공간 정보(코드 경로, 컴포넌트명) 의도적 배제
- 이유: 문제 공간에서 해법 힌트를 넣으면 재귀 분할에서 코드 의존성 발생 → 분리 원칙 위반

**합의된 포맷**: Spec(EARS) + Parent(앵커) + Domain(도메인 태그) + I/O(입출력/조건) + Acceptance Criteria

**Vincent 확정**: "이 이상은 쓰면서 개선하자. 완벽한 처음은 없으니까."

→ 상세: [[04_Action/npdp/issue-decomposition/spec-ticket-format-v1.md]]

## 결정 7: 합류점 N:M 변환 = "로직"이 아니라 "프로토콜"

**Vincent의 질문**: "변환로직을 만들 수 있을까? 결국 이건 현장판단 아닐까?"

**분석 결과**:
- 자동화 가능: 매핑 검증, 중복/미커버/크로스레이어 **감지** (플래그 생성)
- 현장판단: 매핑 자체, 병합/분리 여부, 분리 방법, 우선순위

**결정**: 5-Step 프로토콜 구조 — 코드가 사실 확인과 플래그를 올리고, LLM이 판단하고, 인간이 최종 리뷰. "로직으로 경직시키지 않는다"가 핵심.

→ 상세: [[04_Action/npdp/issue-decomposition/merge-point-protocol-v1.md]]

## B→A 피드백 설계

**Vincent 제안**:
1. A 티켓의 정의는 맞지만 코드 현실과 다른 경우 → B 티켓에 매핑 근거 명시
2. A 분할 자체가 잘못된 경우 (예: "구글 로그인 추가" 티켓이 해법 공간에서는 "로그인 모듈"로 수렴해야 하는 경우) → A 티켓에 코멘트만 남기고 A 트리는 재구조화하지 않음

**이유**: 문제 공간의 트레이서빌리티 보존
