# 합류점 N:M 변환 프로토콜 (v1)

## 핵심 인식

N:M 변환은 "로직"이 아니라 **"프로토콜"** — 현장판단이 핵심이고, 도구의 역할은 판단에 필요한 플래그를 자동으로 올려주는 것까지.

원칙 7번("LLM이 판단, 코드가 검증")이 그대로 적용.

## 자동화 가능한 부분 (코드로 검증, 결정론적)

| 동작 | 방법 |
|---|---|
| 매핑 검증 | spec의 I/O가 참조하는 경로가 실재하는지 코드로 확인 |
| 중복 매핑 감지 | 2개 이상 spec이 동일 컴포넌트에 매핑 → "병합 후보" 플래그 |
| 미커버 감지 | 매핑된 컴포넌트가 없는 spec → "아키텍처 확장 필요" 플래그 |
| 크로스레이어 감지 | spec의 I/O가 여러 레이어(API+DB+UI) 관통 → "분리 후보" 플래그 |

## 인간/LLM 판단 영역 (현장판단)

| 동작 | 판단이 필요한 이유 |
|---|---|
| 매핑 자체 | 도메인→코드 연결은 아키텍처 이해 필요 |
| 병합 여부 결정 | 같은 컴포넌트라도 기능적으로 별개면 분리가 나을 수 있음 |
| 분리 방법 | API/DB/UI로 나눌지, 읽기/쓰기로 나눌지 — 아키텍처 맥락 의존 |
| 우선순위 (P1~P4) | 변경 규모 판단은 코드 이해도에 의존 |

## 변환 프로토콜 (5-Step)

```
[Step 1] LLM이 매핑 초안 생성 (spec → 아키텍처 컴포넌트)
[Step 2] 코드가 매핑 검증 (경로 실재, import 정합성)
[Step 3] 코드가 플래그 자동 생성 (중복/미커버/크로스레이어)
[Step 4] LLM이 플래그 기반으로 병합/분리/확장 판단
[Step 5] 🧑 인간이 결과 리뷰
```

## Status

v1 — 실전 적용하면서 개선 예정. 프로토콜의 세부 단계보다 "코드가 플래그를 올리고, 판단은 LLM+인간"이라는 구조가 핵심.
