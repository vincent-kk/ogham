# 01 — 헌법

seiri의 설계 원칙. **인식론 / 규범 / 베팅** 3층으로 나눕니다. 층을 나누는 이유는, 베팅이 틀렸을 때 헌법 전체가 아니라 **해당 항만 교체**하기 위함입니다.

| 층         | 내용                                  | 성격               |
| ---------- | ------------------------------------- | ------------------ |
| **인식론** | 값 무지 / 형식 인지                   | 방어 가능한 사실   |
| **규범**   | P5 형식 기반 보편성 · P2 소유권 삼분  | 가치 선택          |
| **베팅**   | P1 감가상각 · P3 상태기계 · P4 다이얼 | **검증 대상 가설** |

---

# 층 1. 인식론

> **우리는 개별 저장소의 *값*을 모른다. 그러나 모든 코드베이스가 공유하는 *형식*은 안다.**

| 모르는 것 — **값** (저장소마다 다름)     | 아는 것 — **형식** (코드베이스면 참)             |
| ---------------------------------------- | ------------------------------------------------ |
| 검증 명령(`yarn test`?), 임계치(15? 20?) | 검증 수단이 **존재하고 실행 가능하다**           |
| 언어·프레임워크·빌드 체계                | 코드는 **파일에 담기고 이름·크기·경로를 갖는다** |
| 팀 관례, 네이밍 스타일                   | **기존 스타일이 이미 존재한다**(무엇이든)        |
| 모듈 경계의 위치                         | **공개/비공개 구분이 존재한다**                  |
| 성숙도, 커버리지                         | 변경은 **diff이고 요청에 대응한다**              |
| 커밋 규약                                | 변경은 **기록되고, 이유는 코드에 남지 않는다**   |

**왜 이 정밀화가 필요했나**: "아무것도 모른다"는 강한 무지는 **규칙의 존재 자체를 자기반박**합니다 — 아무것도 모르면 무슨 근거로 보편 규칙을 파는가. 값/형식 분리가 그 모순을 해소합니다.

**저장소를 아는 자는 둘뿐입니다** — 저장소 자신(CLAUDE.md·테스트)과 런타임의 모델(코드를 직접 읽는). seiri는 그 둘에게 소유권을 돌려줍니다.

# 층 2. 규범

## P5. 보편 규칙은 형식에 기대고, 값은 저장소에 위임한다

### 보편성 6층

규칙이 기댈 수 있는 형식의 목록입니다.

| 층          | 공통 형식(전제)                              | 파생 규칙                                         |
| ----------- | -------------------------------------------- | ------------------------------------------------- |
| **L0 파일** | 코드는 파일에 담기고 이름·크기·경로를 갖는다 | 형제 미러링 · 커지면 분할 · 깊은 중첩은 탐색 비용 |
| **L1 참조** | 심볼은 정의되고 참조된다                     | 순환 의존 금지 · 사용처 없는 export               |
| **L2 경계** | 공개/비공개 구분이 존재한다                  | 최소 export · 와일드카드 재수출 금지              |
| **L3 변경** | 변경은 diff이고 요청에 대응한다              | 외과적 변경 · 모든 변경 줄은 요청에 추적          |
| **L4 검증** | 검증 수단이 존재하고 실행 가능하다           | 실패를 보기 전엔 통과가 증거가 아님               |
| **L5 이력** | 변경은 기록되고, 이유는 코드에 남지 않는다   | 결정 맥락 보존                                    |

> **6층에 매핑되지 않는 규칙은 값에 기대고 있다는 신호이므로 기각합니다.**

이 판정은 기계 검사로 집행됩니다 — 각 규칙 문서는 `rests on a property every codebase has: …` 문장을, 두 층 이상에 기대면 `rests on properties every codebase has: …`를 반드시 포함해야 합니다. **수는 기대는 형식의 개수에 맞춥니다** — 두 층에 기대면서 단수로 쓰면 문장이 형식 하나를 숨깁니다([02-ARCHITECTURE.md](./02-ARCHITECTURE.md) 자기검증 절).

**예외 2건**: `context-efficiency`와 `cognitive-discipline`은 코드가 아니라 **세션의 속성**(컨텍스트 유한성, 주장≠증거)에 기댑니다. 예외를 숨기지 않고 문장으로 명시합니다.

### 보편화 3기법

값을 쓰지 않고 규칙을 쓰는 방법입니다.

**① 역할 지칭** — 값 대신 역할로 부릅니다.

> "이 저장소가 지정한 검증 명령을 사용하라. 특정 도구나 러너를 가정하지 마라."

**② 발견 위임** — 값을 주지 않고 값의 출처를 가리킵니다.

> "적용 전에 현행 관례를 발견하라: 형제 파일, 기존 설정, 최근 커밋. 찾은 것을 따르라."

**③ 구조적 불변식** — 언어를 몰라도 참인 사실에만 기댑니다.

> "변경된 모든 줄은 요청에 추적되어야 한다."

**반례로부터 배운 것**: 선례 중 하나는 원칙 문장은 중립인데 **예시가 언어를 고정**했습니다(`npm test`, TypeScript 코드). 자기 문서에 *"트리거를 기술 중립으로 유지하라"*고 써 놓고도 예시에서 무너졌습니다.
⇒ **원칙 문장은 6층 어휘로만, 예시는 의사코드·구조 서술로.**

### 왜 이것이 순응을 낳는가

> **형식 기반 규칙은 저장소와 충돌하지 않으므로 무시할 이유가 없습니다.**

충돌하는 것은 값 기반 규칙입니다. "15케이스 상한"은 어떤 저장소와는 반드시 충돌하고, 충돌하는 순간 유능한 모델일수록 그것을 기각합니다. "사용처 없는 export는 의도를 명시하라"는 어떤 저장소와도 충돌하지 않습니다.

**순응은 압박이 아니라 무충돌성으로 얻습니다.** 이것이 강제 장치 없이 규칙이 작동하는 근거입니다.

## P2. 소유권 삼분

| 소유자     | 소유물                           | 구체물                                                |
| ---------- | -------------------------------- | ----------------------------------------------------- |
| **저장소** | **진실** (무엇이 옳은가)         | 테스트·CI·pre-commit, 그리고 오라클의 정의(CLAUDE.md) |
| **모델**   | **판단** (지금 무엇을 할 것인가) | 계획·구현·리팩토링 결정                               |
| **seiri**  | **맥락** (판단에 필요한 재료)    | 형식 기반 규칙 · 상태 · 주입 · 결정 맥락 보존         |

**선례의 실패를 소유권 침범으로 읽을 수 있습니다**: 승인 게이트는 플러그인이 *판단*을 소유하려는 시도, 차단 훅은 *진실*을 소유하려는 시도입니다. 둘 다 모델이 개선될수록 마찰이 됩니다. seiri는 **맥락만** 소유합니다 — 유일하게 감가상각되지 않는 자산입니다.

### 우선순위 사슬 — 모든 규칙의 첫 줄

```
저장소의 명시 지침 > 저장소의 기존 관례 > seiri의 규칙 > seiri 기본값
```

**근거**: 공식 문서가 *"두 규칙이 모순되면 Claude는 임의로 하나를 고를 수 있다"*고 명시합니다. 충돌 시 임의 선택이므로, **텍스트로 못 박는 것 외에 수단이 없습니다.**

### 오라클 값 비보유

> 검증 명령의 **정의는 저장소의 CLAUDE.md가 소유**합니다. seiri는 그 값을 탐지·등록·주입하지 않습니다.

세 가지 이유입니다.

1. **자기 위반** — 값을 관리하는 순간 저장소 진실의 **사본**을 소유하게 되고, 사본은 드리프트합니다.
2. **중복** — 저장소 CLAUDE.md는 하니스가 이미 모델에 주입합니다.
3. **설정 불가능성** — 이 값은 저장소마다 다르고 저장소에서 정할 문제입니다.

seiri가 갖는 것은 규칙 문서 안의 원칙 문장 하나뿐입니다 — _"저장소의 검증 수단을 따르라."_

게이트 스캐폴드를 제공할 때도 **placeholder 골격만** 생성하고, 값은 모델이 CLAUDE.md에서 전사합니다.

### 오라클 부재는 1급 상태

CLAUDE.md도 테스트도 없는 저장소에서, seiri의 유일한 자동 개입은 **"오라클을 세워라"는 촉구**입니다. 값 주입이 아니라 **존재의 촉구**이므로 P2를 위반하지 않습니다.

### 강한 통제는 저장소에 위임하는 것이 더 강하다

플러그인 훅은 *플러그인이 설치된 세션*만 구속합니다. 저장소 게이트(CI·pre-commit)는 **모든 행위자**(사람이든 어떤 에이전트든)를 구속하고, 저장소별로 보정됩니다.

seiri는 게이트를 **소유하지 않고 스캐폴드**합니다 — 요청 시 생성을 돕고, 이후는 저장소 소유입니다.

# 층 3. 베팅

**이 층은 검증 대상 가설입니다.** 틀리면 해당 항만 교체합니다.

## P1. 감가상각 테스트 — capability와 reliability를 가른다

모든 장치에 묻습니다: **"모델이 두 배 똑똑해지면 이 장치의 가치는 오르는가, 내리는가?"**

| 감가상각 — **capability 대체** | 불감가상각 — **reliability 구매**     |
| ------------------------------ | ------------------------------------- |
| 키워드 자동 감지               | **저장소 소유 게이트**(CI·pre-commit) |
| 무차별 승인 게이트             | 결정 맥락 보존                        |
| 스텝바이스텝 절차 지시         | 상태 지속 · 깨끗한 컨텍스트 예산      |

**중요한 구분**: 초기 설계는 강제 일반을 "모델이 좋아지면 무의미"로 몰았습니다. 틀렸습니다. 강제의 진짜 가치는 능력 보완이 아니라 **열화 조건의 신뢰성**입니다 — 컴팩션 직후 망각, 긴 세션의 컨텍스트 압박, 그리고 **다중 행위자**(다른 사람·다른 도구가 커밋하는 코드). 이 실패 모드들은 자기회귀·유한 컨텍스트·멀티액터의 **구조적 속성**이라 모델 개선으로 사라지지 않습니다.

### 보장 수준 3등급 — 정직한 표기

| 등급       | 수단                             | seiri                                                 |
| ---------- | -------------------------------- | ----------------------------------------------------- |
| `prompted` | 규칙 문서·스킬 본문              | **주 산출물**                                         |
| `observed` | `InstructionsLoaded` 훅 로깅     | 측정용                                                |
| `enforced` | PreToolUse 훅 · 권한 · 저장소 CI | **저장소 소관** (스킬 `disallowed-tools`만 부분 해당) |

공식 문서가 확인합니다 — _"규칙과 CLAUDE.md는 강제 설정이 아니라 컨텍스트다. 무엇을 하든 차단하려면 PreToolUse 훅을 쓰라."_ seiri는 이를 숨기지 않고 등급으로 표기합니다.

## P3. MCP는 도구상자가 아니라 상태기계

**컨텍스트로 나가는 것은 도구 스키마가 아니라 상태의 렌더입니다.** 복잡성은 코드(상태기계) 안에 격리되고, 주입은 짧은 요약으로 고정됩니다.

- 상태 3종: 설정 / 규칙 배포 상태 / 세션 플래그
- **도구상자화 금지** — 도구는 상태를 읽고 쓰는 최소만. 코드 분석·검색 도구는 하니스가 이미 제공합니다.
- MCP 도구는 **스키마 자체가 컨텍스트 비용**이므로 규모 목표에 계상합니다.

**측정 신호**: `InstructionsLoaded` 훅이 *어떤 지시 파일이 언제 왜 로드됐는지*를 로깅합니다. seiri가 **자기 효능에 대한 오라클**을 갖는 유일한 공식 경로입니다 — P2에 따라 진실은 저장소가 소유하지만, seiri 자신의 효능은 seiri가 측정해야 합니다.

## P4. 통제는 스위치가 아니라 다이얼

`advisory`(기본) / `standard` / `strict` 3단.

| 축                   | advisory | standard | strict      |
| -------------------- | -------- | -------- | ----------- |
| 주입 컨텍스트 블록   | 최소     | 기본     | 기본 + 선택 |
| 자동 스킬 발동 폭    | 최소     | 기본     | 넓게        |
| 게이트 스캐폴드 권유 | 안 함    | 제안     | 적극 제안   |
| 세션 신호 훅         | **침묵** | 활성     | 활성        |

**적용 범위**: 다이얼은 **주입(렌더)과 세션 훅 신호만** 바꿉니다. 배포된 규칙 문서 자체는 바꾸지 않습니다 — 해시 드리프트 감지와 충돌하기 때문입니다.

**저장은 2계층입니다** — 기준선 `.seiri/config.json`(커밋 대상, setup 표면 전용)과 런타임 밸브 `.seiri/runtime.json`(비추적, 대화로 조작). 저장소가 선언하는 개입 강도는 진실이라 뻑뻑하게(P2), 세션의 조작은 값싸게 — 한 파일이 둘을 겸하면 다이얼 조작이 워킹트리를 더럽히고 낮춘 값이 커밋에 섞여 기준선을 침식합니다. 구조는 [02-ARCHITECTURE.md](./02-ARCHITECTURE.md) 다이얼 절.

**왜 다이얼인가**: 강도를 하드코딩하면 모든 사용자가 같은 마찰을 받고, 통제를 구조에 박으면 모델 개선을 따라가지 못합니다. 다이얼이면 **모델이 좋아질 때 사용자가 내리면 됩니다 — 재설계 없이.** P1의 실행 장치입니다.

**문구 강도(MUST/SHOULD) 조절은 폐지 확정했습니다**(2026-07-24). 배포 문서와 주입 문구가 서로 다른 강도를 갖게 되는 이중 강도 문제 — 다이얼은 문구가 아니라 **블록과 신호의 유무**를 바꿉니다.

# 역할 밖 목록

**코드보다 먼저 고정합니다.** 대형화는 개별 기능이 나빠서가 아니라 **거절 기준이 없어서** 발생합니다.

```
seiri is responsible for:
  code authoring quality, code review discipline, development methodology.

seiri is NOT responsible for:
  architecture enforcement (filid), agent orchestration (omc),
  task decomposition (imbas), knowledge management (maencof),
  external CLI delegation (cennad),
  notifications, status display, installation tooling,
  benchmarking infrastructure, marketing materials.

seiri does NOT own:
  truth (the repo owns it — tests, CI, CLAUDE.md),
  judgment (the model owns it).

When a proposed feature is not in the first list, it does not enter this
repository — even if it is good.
```

**도출**: 뒤 두 절은 P2 소유권 삼분에서 직접 나옵니다 — 진실 소유 장치(차단 훅·오라클 값 보유)와 판단 소유 장치(무차별 게이트)는 정의상 역할 밖입니다.

**근거**: 한 선례는 자기 에이전트 19개 **전부**에 *"You are not responsible for…"*를 썼으나 **플러그인 자신에게는 쓰지 않았습니다.** 그 결과 알림 시스템(14,869줄), 상태 표시 HUD(11,163줄), 발표 슬라이드(5,989줄)가 들어왔습니다. 각각은 그 자체로 나쁘지 않습니다. 사후 판단에 맡기면 개별 판단은 언제나 "이건 유용한데?"로 끝납니다.
