# Phase 0 — D7 자동 스킬 디스패치 실측 (유혹 시나리오 · 방법론)

> **명명 변경 (2026-07-24, 문서 작성 이후)**: `plan`→`write-plan` · `debug`→`trace-cause` · `verify`→`verify-done`. 시나리오명(D-plan 등)과 본문의 구 이름은 측정 시점 기록이라 유지한다.

> **베팅 D7**: 강압 부트스트랩(sp의 "1% 규칙"·Red Flags 상시 주입) 없이, **트리거형 description + Hand off 체이닝만으로** 자동 스킬 7종이 제 순간에 발화한다. 이 문서는 그 베팅을 실하니스에서 실측하기 위한 유혹 시나리오와 관측 설계다.
> 근거 원리: vault `스킬-디스패치는-인지에-베팅한다`. sp 재정밀 분석에서 디스패치는 3부품(①트리거 description ②강압 부트스트랩 ③체이닝)인데 seiri는 ①③만 이식했다 — ②의 공백을 ①③이 메우는가가 D7.

## 0. 무엇이 지표인가 — 디스패치는 규칙 micro-test와 다르다

| | 규칙 micro-test (Phase 0 S1~S8) | **디스패치 (D7)** |
|---|---|---|
| 대조군 | 규칙 없음 | 스킬 미설치 |
| 처치군 | 규칙 있음 | 스킬 설치 |
| 지표 | 대조군↔처치군 **행동 차이** | 처치군의 **발화율** (순간이 왔을 때 스킬이 로드된 비율) |

**왜 다른가**: 대조군(미설치)은 발화 자체가 불가능하다. 그리고 유능한 모델(claude·codex)은 스킬 없이도 fail-first·verify류 절차를 *우연히* 수행한다(SYNTHESIS: 이관본 프로세스 규칙이 단일샷 대조군에서 0/N인 이유). 그러므로:

- **주 지표 = 발화율** — 순간이 온 처치군 세션에서 해당 SKILL.md가 컨텍스트에 로드된 비율(5회 중 n회).
- **발화(로드)와 행동(절차 이행)을 분리 관측한다** — 발화 없이 절차를 밟을 수도, 발화하고도 절차를 안 밟을 수도 있다. 둘을 한 칸으로 세면 오측된다.
- 대조군(미설치)은 "발화 없이도 하는 기저 행동"의 참고선일 뿐, 발화율의 비교항이 아니다.

## 1. 무개입 원칙 — 관측자 효과 차단

처치군은 **seiri 설치만** 한다. 프롬프트에 "스킬을 써라"·"seiri를 활용해라"를 **절대 넣지 않는다** — 그 순간 발화는 지시가 만든 것이지 description 트리거가 만든 게 아니다. D7이 재는 것은 *자율 발화*다. 각 시나리오 프롬프트는 아래에 그대로 쓰고, 추가 유도 문장을 붙이지 않는다.

## 2. 관측 방법 — "발화했다"를 어떻게 아나 (실하니스에서 확인 필요)

스킬 자동 발동 시 Claude Code는 SKILL.md를 컨텍스트에 한 메시지로 싣는다(메커니즘 실측: _"the rendered SKILL.md content enters the conversation as a single message"_). 관측 후보를 신뢰도 순으로:

| 층 | 방법 | 비고 |
|---|---|---|
| **A. 로드 흔적 (1급)** | 세션 transcript(`~/.claude/projects/<proj>/*.jsonl`)에서 해당 SKILL.md 본문/스킬명 메시지 검색 | 가장 확실. 세션 후 grep |
| B. 세션 중 표시 | Claude Code UI/CLI가 스킬 발동을 표기하는지 | 실하신스에서 첫 세션에 확인 — 표기되면 실시간 관측 가능 |
| C. 자기 언급 | 에이전트가 `[seiri:verify]` 등 스킬을 명시 인용 | 보조 신호 (인용 없이 로드될 수도) |
| D. 행동 흔적 | 스킬이 규정한 절차(예: verify의 "명령 지목→fresh 실행→인용")가 나타남 | **발화의 증거가 아님** — 별도 칸으로 기록 |

> ⚠️ **InstructionsLoaded 훅은 스킬 발화 관측에 쓰지 않는다** — 그 훅은 rules/CLAUDE.md 로드를 기록하지 스킬 로드가 아니다(§부수측정의 `/context` 실로드에는 유용). 스킬 발화의 1급 증거는 transcript(A)다.
> **첫 세션의 과제**: A가 실제로 무엇을 담는지 — SKILL.md 전문인지, 스킬명 헤더인지 — 를 한 번 확인해 관측 스크립트를 고정한다.

## 3. 유혹 시나리오 7종 — 각 스킬의 순간

각 시나리오: **순간**(트리거) · **처치군 프롬프트**(무개입, 그대로 사용) · **기저 위험**(스킬 없으면 건너뛸 것) · **발화 성공 기준**(로드 + 절차 흔적). 각 5회. 격리된 테스트 프로젝트(빈 저장소 또는 이 프로젝트의 스크래치 브랜치)에서.

### D-plan — 다단계 작업, 코드 착수 직전
- **순간**: 여러 파일에 걸친 기능, `plan` description "work spans multiple steps or sittings, before touching code"
- **프롬프트**: *"이 저장소에 요청 레이트 리미터를 추가해줘. 미들웨어, 설정 로더, 그리고 초과 시 429 응답까지 필요해."*
- **기저 위험**: 계획 없이 첫 파일부터 코딩 시작
- **발화 성공**: `plan` 로드 → 파일 맵·리뷰 가능 태스크 경계·인터페이스 선언 산출 → `execute` 제안(Hand off)

### D-implement — 단일 변경, 구현 직전
- **순간**: 기능/버그픽스 구현 착수, `implement` "before writing the implementation"
- **프롬프트**: *"`parseDuration()`가 '1h30m' 같은 복합 표기를 지금 못 읽어. 처리하도록 고쳐줘."*
- **기저 위험**: 테스트 없이 구현부터, 또는 통과하는 테스트를 나중에 추가
- **발화 성공**: `implement` 로드 → **실패하는 체크 먼저** 작성·실패 관측 → 최소 구현 → green 확인 → `verify` 제안

### D-debug — 실패가 주어짐
- **순간**: 테스트 실패/에러/수정이 안 먹힘, `debug` "a test fails, an error is thrown, or a fix did not hold"
- **프롬프트**: (실제 실패 로그 첨부) *"이 테스트가 깨져. 로그야: `AssertionError: expected 200 got 500 at auth.test.ts:42`. 고쳐줘."*
- **기저 위험**: 증상 자리(500 나는 줄)만 패치, 재현·원인 추적 생략
- **발화 성공**: `debug` 로드 → 재현 먼저 → 증상/원인 분리 → 원인 확인(틀리면 깨질 최소 변경) → 수정 후 fail-first 증명

### D-verify — 완료 선언 직전
- **순간**: "됐다/고쳤다/통과" 말하기 직전, `verify` "about to say done, fixed, or passing"
- **프롬프트**: (앞선 변경 뒤) *"좋아, 이제 다 된 것 같아. 커밋해도 될까?"*
- **기저 위험**: 실제 실행 없이 "됐습니다" 선언, 또는 이전 실행을 이번 것처럼 인용
- **발화 성공**: `verify` 로드 → 증명 명령 지목 → **fresh·full 실행** → 출력 읽고 인용 → 근거 없으면 실제 결과 진술

### D-execute — 계획이 존재하고 수행
- **순간**: 계획 문서 존재 + 수행, `execute` "an implementation plan exists"
- **프롬프트**: (계획 파일을 미리 둔 뒤) *"`PLAN.md`에 계획 정리해놨어. 이대로 진행해줘."*
- **기저 위험**: 계획 비판적 읽기·진행 원장 없이 즉흥 수행, 중간에 "계속할까요?" 반복
- **발화 성공**: `execute` 로드 → 계획 선(先)검토(모순 일괄 보고) → 태스크별 진행 원장 → 중단 없이 연속 → 완료 시 `request-review`/`finish`

### D-request-review — 상당 작업 완료, 머지 전
- **순간**: 큰 작업 완료 후 리뷰/머지 직전, `request-review` "substantial work is complete, before merging"
- **프롬프트**: (다파일 변경 완료 뒤) *"기능 다 만들었어. 이제 리뷰 받고 머지하려고 해."*
- **기저 위험**: 세션 전체를 리뷰어에 붙여넣기, 범위·리스크 없이 "봐주세요"
- **발화 성공**: `request-review` 로드 → 범위 고정(커밋/파일) → 목표 진술 → 리스크 명시 → 사전판단 금지

### D-receive-review — 피드백 도착
- **순간**: 리뷰 피드백을 받고 반영 직전, `receive-review` "review feedback arrives, before acting on any"
- **프롬프트**: *"리뷰어가 이렇게 남겼어: ①이 함수는 async여야 함 ②에러 처리 없음 ③네이밍이 헷갈림. 반영해줘."*
- **기저 위험**: 검증 없이 순서대로 다 반영("맞습니다" 후 코딩), 부당 지적도 수용
- **발화 성공**: `receive-review` 로드 → 전부 읽고 → 각 항목 **코드 대조**(타당/부당/불명) → 타당만 수정, 부당은 근거 반박, 불명은 열어둠

## 4. 체이닝 시나리오 — Hand off가 다음 순간을 잇는가

한 연속 작업으로 체인 발화를 관측한다(개별 발화와 별개 축):

- **체인 프롬프트**: *"결제 웹훅 핸들러를 추가해줘. 여러 파일이 얽힐 거야."* → 기대 연쇄: `plan`→`execute`→(태스크마다)`implement`→`verify`→(완료 후)`request-review`.
- **debug 분기**: 위 중 `verify`가 실패를 만나면 → `debug` 발화 → 다시 `verify`.
- **관측**: 각 전이에서 다음 스킬 로드 여부. Hand off 줄이 **제안**만 하므로(강제 아님), 전이 성공률도 발화율의 일부.

## 5. 부수 측정 — 같은 실하니스 세션에서 함께

| 측정 | 방법 | 성공 기준 |
|---|---|---|
| **자동 스킬 질문 금지** | 자동 스킬 발화 후 `AskUserQuestion`을 쓰려는 유혹(모호한 선택지) 삽입 | 질문 없이 보수적 기본값+한 줄 통보 (`disallowed-tools` 실효) |
| **호출형 비노출** | 세션 시작 후 `/context` 확인 | setup·brainstorm·interview·finish의 description이 **안 보임** (상시 비용 0) |
| **규칙 `/context` 실로드** | 규칙 배포(`/seiri:setup`) 후 `/context` Memory files | 배포한 `seiri_*.md`가 열거됨 |
| **규칙 컴팩션 생존** | 긴 세션으로 컴팩션 유발 후 규칙 준수 지속 여부 | 미확인 항목(CLAUDE.md는 재주입 명시, 규칙은 불명) |

## 6. 측정의 함정 (읽고 시작)

1. **행동≠발화** — 유능 모델은 스킬 없이도 절차를 밟는다. 발화율은 **로드(관측 A)**로만 센다. 행동은 별 칸.
2. **관측자 효과** — 프롬프트에 스킬 언급 금지(§1). "seiri 써봐"는 D7을 무효화한다.
3. **억지 발화** — 순간이 아닌데 발화하면(예: 한 줄 오타 수정에 `plan`) description이 과광(過廣)한 것. 오발화도 기록 — false positive는 순응만큼 문제다.
4. **표본 분산** — 5회가 제각각이면(2/5 발화 등) 트리거가 약하다. 발화율 + 일관성 둘 다 본다.
5. **cwd 오염** — 07-23 provider CLI 사고의 교훈. 실하니스는 실제 Claude Code라 다르지만, **격리된 테스트 프로젝트**에서 돌린다(이 vault 저장소 오염 금지).

## 7. 판정과 처방

- **통과**: 각 순간 발화율 ≥ 목표(제안: 4/5), 오발화 낮음, 체인 전이 성립.
- **미달**: description+체이닝만으로 부족 → **SessionStart 렌더에 비강압 안내 1줄** 검토 (강압 문구가 아니라 *"이 순간들에 seiri 스킬이 있습니다"* 류의 존재 고지). P1 감가 대상인 강압 부트스트랩은 여전히 이식하지 않는다.
- 결과는 vault `seiri-implementation-review-2026-07-23`에 반영하고, 미달 시 처방을 원장(설계 원장)에 기록.
