## 0. 왜 D7 하니스를 그대로 쓸 수 없나

[d7-dispatch.md](./d7-dispatch.md)의 프록시 하니스는 **신선한 서브에이전트 + 정확한 프롬프트 + transcript grep**이었고, 그것이 D7을 33/35로 결론냈다. 그 하니스를 여기 쓸 수 없다:

> **프록시의 정의적 속성(신선·소컨텍스트·단발)이, 확장 트랙이 겨냥하는 조건의 정확한 반대다.**

긴 세션·컴팩션 후·연쇄 실패 중·위임 — 이것들은 프록시가 재현할 수 없어서 선제 배선을 택한 것이다. 프록시로 재보면 "잘 되네"가 나오는데, 그 답은 이미 D7이 준 답이지 새 질문의 답이 아니다.

그래서 하니스를 **두 층으로 가른다** — 성격이 다른 두 질문이기 때문이다.

| 층               | 묻는 것                            | 성격      | 비용        | 표본                |
| ---------------- | ---------------------------------- | --------- | ----------- | ------------------- |
| **T1 배선 검증** | 실하니스에서 **배관이 작동하는가** | 결정론적  | 1세션 ~30분 | 각 1회 pass/fail    |
| **T2 사후 관측** | 대컨텍스트에서 **값을 하는가**     | 사건 기반 | 실사용 누적 | 기회주의적 사례수집 |

T1에 통계가 필요 없는 이유: 배선은 되거나 안 된다. T2에 통계를 쓸 수 없는 이유: 조건을 만들 수 없고 빈도가 낮다 — 그래서 **비율이 아니라 사례**를 모은다.

### 1.1 baseline 은 advisory 로 두고 밸브로 강도를 테스트하라 (실측 정정)

T1 실측의 스크래치는 **baseline(`config.json`) 자체가 standard** 였고, 그래서 `clear` 가 advisory 가 아니라 standard 로 복귀했다. 결과 문서는 이를 "T2 advisory 기준선 전제가 깨졌다"로 읽었으나 — **그 판정의 근거는 스크래치 아티팩트다.**

- **~~seiri 의 배포 기본값은 advisory 다~~ → 2026-07-24 `standard` 로 변경** (`DEFAULT_INTERVENTION='standard'` · `SILENT_INTERVENTION='advisory'`, 사용자 확정). 새 프로젝트는 이제 **standard 로 시작**하고(켜면 workflow chain on), advisory 는 opt-out floor 로 남는다. **아래 '기본값 advisory' 분석은 그 이전 기준이며, 이 변경으로 §1.1 절차와 §4 T2 기준선 전제는 재결정 대상이 됐다.**
- 스크래치가 standard 였던 것은 **항목 5~8(standard↑ 에서만 작동)을 테스트하려고 baseline 을 조작했기 때문**이다. `config` action 의 `set` 은 `runtime.json` 만 건드리므로, baseline 을 올리려면 setup 표면을 거쳤어야 했다 — 그게 `clear` 가 advisory 로 못 간 이유다.
- **올바른 절차**: baseline 은 advisory 로 두고, 강도가 필요한 항목은 **runtime 밸브(`set standard`/`set strict`)로** 테스트한다. 그러면 `clear` 가 밸브를 떨구고 advisory 로 깨끗이 복귀해, T2 기준선이 보존된다.

~~따라서 **T2 의 advisory 기준선 전제는 유효하다** — 실사용 저장소가 기본값(advisory)이면.~~ **⚠️ 2026-07-24 재역전**: 배포 기본값이 `standard` 로 바뀌어 이 전제의 조건("기본값 advisory")이 무너졌다. 실사용 저장소는 이제 기본 standard 이므로, T2 무비용 기준선(advisory)을 만들려면 **관측 시 명시적으로 `runtime set advisory` 로 내려야** 한다 — 또는 기준선 자체를 standard 로 재설계한다. **T2 착수 전 결정 필요.**

| **S3**   | 테스트 작업을 했다                             | 조건부 규칙이 실제로 그 순간 로드됐나                                      |

### 기록 규율 (D7에서 계승)

- **주입 ≠ 행동**(D7 §6.1) — "신호가 떴다"와 "행동이 바뀌었다"는 다른 칸이다. 한 칸으로 세면 오측된다.
- **오발화도 기록**(D7 §6.3) — false positive 는 미발화만큼 중요하다. 특히 D13 은 의도된 red 와 충돌할 수 있게 설계돼 있다.
- **관측자 효과**(D7 §1) — "지금 신호 뜨나 보자"고 조건을 유도하지 않는다. 유도하면 그건 T1이지 T2가 아니다.

기록 위치: 이 문서 말미 §6 로그. 사건당 한 줄.

## 3. 처리방향 — 사전 등록된 판정 규칙

**관측 전에 정해둔다.** 결과를 보고 기준을 정하면 무엇이 나와도 "역시 잘 만들었다"로 끝난다.

| 관측된 것                                | 해석                          | 처리                                                                              |
| ---------------------------------------- | ----------------------------- | --------------------------------------------------------------------------------- |
| D13 오발화(의도된 red)가 적중보다 잦다   | N=3 이 짧거나 둘을 구분 못 함 | 1차 N 상향(3→5). 그래도 잦으면 **훅 제거** — 판단을 흐리는 신호는 없느니만 못하다 |
| D13 적중했는데 행동 불변                 | 주입이 안 읽힌다              | 문구 조정 1회 → 그래도 불변이면 **제거**(값 없는 상시 비용, P1)                   |
| D13 이 한 번도 안 뜸                     | 조건이 드물다 / 게이팅 오류   | T1 #7·#8 로 배선 재확인 → 정상이면 **유지**(보험료가 싸다는 뜻)                   |
| D12 주입 후에도 서브에이전트가 규칙 무시 | 위치 지목만으로 부족          | 위임 문장(스킬)만 남기고 **훅 제거** 검토 — 훅은 상시, 문장은 순간                |
| 렌더·신호가 거슬린다                     | 다이얼이 높다                 | **코드 변경 없이 다이얼 하향** — 설계된 opt-out 이 작동하는 것                    |
| 컴팩션 후 규칙 미준수                    | 하니스 한계                   | InstructionsLoaded 재등록으로 재측정 → 결과에 따라 SessionStart 재주입 검토       |
| S3 조건부가 로드 안 됨                   | 패턴 문제 or 배포 문제        | 패턴 점검(gitignore 문법·루트 상대·후행 `/**`) → 안 되면 **무조건 로드로 철회**   |

### 두 가지 원칙

**① 제거가 1급 선택지다.** 선제 배선은 "값이 없으면 뺀다"를 전제로 지어졌다. 지었으니 지킨다는 관성은 선제 배선을 정당화한 논리 자체를 배신한다.

**② 다이얼 하향과 코드 제거는 다른 층이다.** 하향은 _사용자의 선택_(이 저장소에선 시끄럽다), 제거는 _설계 판정_(어디서도 값을 못 한다). 하향으로 해결되는 문제를 제거로 처리하면 다른 저장소의 값까지 버린다.
