# YouTube 자막/콘텐츠 추출 MCP 서버 — 더 완성도 있고 안전하게 만드는 방법 (2026)

> 기술스택: Node.js + TypeScript
> 작성 일자: 2026-06-07
> 근거: 현재 구현 분석([current-implementation-analysis.md](./current-implementation-analysis.md)) + codex/antigravity 교차검증 리서치

## 0. 목표와 현재 방식의 한계

현재 구현은 "**InnerTube 직접 호출 + 정규식 HTML 스크래핑 + ANDROID 클라이언트 위장 + 수작업 protobuf 인코딩**"으로, 의존성 없이 동작하는 완성도는 높지만 다음이 취약하다.

- 정규식 스크래핑 → YouTube 레이아웃 변경 시 **무증상 실패**
- 하드코딩 클라이언트 버전 → YouTube의 강제 업데이트로 **주기적 만료**
- poToken 강제화 흐름에 단일 ANDROID 위장으로 대응 → **지속 가능성 의문**
- 단일 경로 → 실패 시 폴백 없음
- stdout 로깅, 테스트 부재 등 운영 품질 이슈

핵심 방향: **"하나의 영리한 스크래퍼"가 아니라 "교체 가능한 provider 기반 추출 서비스"로 재설계한다.**

## 1. 자막/콘텐츠 획득 방법론 비교

| 방법 | 견고성 | 유지보수성 | ToS/법적 리스크 | poToken 대응 | 2026 적합성 |
|------|--------|-----------|----------------|--------------|-------------|
| (a) InnerTube 직접 호출(현재 방식) | 중(잘 짜면) / 하(수작업) | 나쁨 | 높음(비공식 API) | 전부 직접 부담(버전·visitorData·토큰·응답변형) | 보통 |
| (b) 공식 Data API v3 `captions.download` | 높음(인가된 영상) | 높음 | **가장 낮음** | 무관(OAuth) | **낮음** — 본인 소유/인가 영상만, 호출당 200 quota |
| (c) **youtubei.js (YouTube.js)** | InnerTube 추상화 최선 | 좋음(활발히 관리) | 높음(비공식) | `po_token`/`visitor_data`/client type 옵션 지원, BgUtils 결합 | **높음 — 권장 primary** |
| (d) youtube-transcript류 | 스크립트용으론 OK, 프로덕션 약함 | 가변 | 높음 | 대부분 없음, 차단에 취약 | 낮음(포크 `@danielxceron/...`만 부분 작동) |
| (e) yt-dlp 서브프로세스 | **실전 커버리지 최고** | 중(외부 바이너리 수명) | 높음(추출기 우회) | PO-token provider 플러그인 가장 완벽 | 높음(opt-in 폴백) |
| (f) 레거시 timedtext API | 캡션 URL 알면 단순·빠름 | 낮음(비공식) | 높음 | 없음, poToken 없이 대부분 403 | **낮음**(사실상 차단 추세) |
| (g) Whisper/STT 폴백 | 오디오만 얻으면 견고 | 중~높음(인프라 비용) | 권리/저장/동의 의존 | 캡션 poToken 회피(오디오는 별도 차단) | **최후 보루(권장)** |

요점:
- **공식 API는 범용 불가** — 타인 영상 자막은 받을 수 없다. "본인 채널 추출" 특수 모드로만 의미.
- **youtubei.js를 코어로** 두면 수작업 protobuf/regex/클라이언트 위장의 churn을 라이브러리가 대신 흡수한다.
- **단일 방법에 의존하지 말 것** → 다계층 폴백.

## 2. YouTube 봇 차단 / poToken 동향 (2025-2026)

- **poToken(Proof of Origin Token) 강제화 확산**: BotGuard(웹)/DroidGuard(모바일) 챌린지로 토큰을 계산해야 하며, 누락·위조 시 403 또는 "Sign in to confirm you're not a bot". 이제 niche가 아니라 GVS/플레이어/자막 요청에 광범위.
- **poToken은 IP·visitorData와 암호학적으로 결합** → 토큰 생성 IP와 요청 IP가 달라지면 무효. **세션 단위 IP 고정** 필요. 토큰 캐시는 `{visitor/session/video/case}` 범위로 세분.
- **클라이언트별 현황**:
  - `WEB`: 브라우저에 가깝지만 BotGuard 노출 최대.
  - `MWEB`: yt-dlp가 토큰 provider와 함께 현재 선호.
  - `ANDROID`/`iOS`: 여전히 유효하나 **강제 업데이트로 구버전 차단 잦음** → 최신 버전 추적 필요.
  - `TVHTML5`: 과거엔 토큰 불필요했으나 **2025년 이후 DRM/암호화로 난이도 급상승**.
  - `WEB_EMBEDDED`: 제한적 메타데이터/플레이어용, 범용 자막 해법 아님.
- **BgUtils(`bgutils-js`, LuanRT)**: JSDOM 위에서 BotGuard 스크립트를 실행해 헤드리스 브라우저 없이 순수 Node에서 poToken+visitorData 생성. 성능상 유리.
  - **단, 채택 강도는 논쟁** — 아래 §8 의견 차이 참고. **mandatory로 번들하지 말고 `PoTokenProvider` 인터페이스로 추상화("bring your own provider")** 하는 것이 공개 패키지엔 안전.

## 3. 안전성·견고성 엔지니어링

- **정규식 → 안전한 JSON 추출**: watch HTML을 읽어야 하면 `ytInitialPlayerResponse`/`ytInitialData` 경계를 **balanced-brace 파서**로 잘라 `JSON.parse`, 이후 **zod로 검증**. 정규식 스크래핑 폐기.
- **무증상 실패 방지 → 타입드 에러 taxonomy**: `NoCaptions`, `Blocked`, `PoTokenRequired`, `VideoUnavailable`, `AgeRestricted`, `ParseChanged`, `RateLimited`. `|| ''` 식 조용한 빈값 금지, 명시적으로 throw.
- **재시도**: transient(429/5xx/네트워크 리셋)에만 **지수 백오프 + 지터**. 파싱 실패는 재시도 금지.
- **캐싱**: 자막은 정적 데이터. `{videoId, lang, includeAuto, providerVersion}` 키, 긴 TTL(7~30일). 음성/세션/visitorData/플레이어 config는 분리 캐시. 부정 결과(negative)는 짧은 TTL.
- **동시성/예산**: 호스트별 동시성 제한 + 전역 요청 예산(`p-limit`/`bottleneck`).
- **프록시**: 명시적 config로만. 차단 회피용 순수 로테이션은 정책/법적 리스크 → rate limit+캐싱 우선. 대규모 운영 시 데이터센터 IP는 즉시 차단되므로 **주거용 프록시 + 세션 IP 고정**(poToken-IP 결합 때문) 필요.
- **관측성**: stdio 서버는 **stdout 절대 금지**(JSON-RPC 채널). `console.error` 또는 `pino`(destination 2/stderr). 쿠키/OAuth/poToken/전체 자막 텍스트는 기본 로깅 제외. 어떤 client 전략이 노후화 중인지 보이도록 client name/version/case/실패코드 텔레메트리 수집.

## 4. MCP 서버 아키텍처 모범사례

- **SDK**: `@modelcontextprotocol/sdk`의 high-level **`McpServer`** + 모든 툴 입력에 **zod 스키마**.
- **트랜스포트**: 로컬(Claude Desktop/Cursor 등)은 **stdio**(포트 없음 → 안전). 원격/멀티유저는 **Streamable HTTP**(2026 표준, 단일 엔드포인트). SSE는 deprecated/하위호환용.
- **구조화 출력**: 텍스트만이 아니라 structured content 반환.
  ```ts
  {
    videoId, title, author, language, isAutoGenerated,
    source: "youtubei" | "timedtext" | "yt-dlp" | "data-api" | "stt",
    segments: [{ startMs, durationMs, text }],
    chapters: [{ startMs, title }],
    warnings: []
  }
  ```
- **툴 분리**: `youtube_transcript_get`, `youtube_video_get`, `youtube_chapters_get`, `youtube_comments_get`, `youtube_thumbnail_get`, `youtube_audio_transcribe`(설정 시에만 활성).
- **테스트 전략**:
  - 단위: URL/ID 파싱, 언어 폴백, 광고/챕터 스트립, 응답 정규화, 타입드 에러 매핑.
  - **fixture 테스트**(핵심): 녹화된 watch HTML / InnerTube JSON / timedtext JSON3·XML / yt-dlp JSON으로 각 provider 검증(`nock`/`msw`).
  - 계약 테스트: MCP SDK 클라이언트로 stdio 툴 검증.
  - 통합 테스트: env로 opt-in, rate-limited, 매트릭스(공개 자막/자동 자막/자막 없음/종료된 라이브/Shorts/비공개).
  - 정규화된 출력에만 스냅샷(원본 YouTube 응답엔 금지).

## 5. 권장 아키텍처 (provider 추상화 + 다계층 폴백)

```
요청(videoId) → 캐시 확인
  └ miss → [1] youtubei.js (+ 옵션 PoTokenProvider/BgUtils)
            └ 실패(403/봇) → [2] yt-dlp 서브프로세스 (opt-in)
                              └ 실패 → [3] STT 폴백(opt-in): yt-dlp로 오디오만 → faster-whisper/Whisper API
                                        └ 최종 실패 → 타입드 에러
  └ 성공 → 정제·정규화 → 캐시 저장 → 반환
```

모듈 구성(관심사 분리 — 획득/정규화/후처리/전송/관측을 독립적으로):
- `providers/TranscriptProvider.ts` — 공통 인터페이스
- `providers/YoutubeJsProvider.ts` — primary
- `providers/TimedTextProvider.ts` — 단순 캡션 폴백(주력 금지)
- `providers/YtDlpProvider.ts` — opt-in 서브프로세스
- `providers/DataApiProvider.ts` — 공식 OAuth 모드(본인 채널)
- `providers/SttProvider.ts` — opt-in Whisper 폴백
- `youtube/SessionManager.ts` — visitorData/캐시/client 전략/옵션 poToken provider
- `youtube/CaptionSelector.ts` — 언어 폴백, 수동 vs 자동 우선
- `postprocess/AdStripper.ts` — 챕터/광고 마커 정책(획득 로직과 분리)
- `mcp/server.ts` — 툴만, YouTube 파싱 없음
- `observability/logger.ts` — stderr-safe 구조화 로깅
- `errors.ts` — 타입드 에러 taxonomy

라이브러리 선택:
- InnerTube: **`youtubei.js`**
- poToken(옵션): **`bgutils-js`**
- MCP: **`@modelcontextprotocol/sdk`** (+ Streamable HTTP 필요 시)
- 스키마: **`zod`**
- 로깅: **`pino`**
- 캐시: `lru-cache`(소규모) / SQLite·Redis(대규모)
- 동시성/레이트: `p-limit` 또는 `bottleneck`
- 테스트: `vitest` + `nock`/`msw`
- 서브프로세스: `execa`(yt-dlp)
- STT: `whisper.cpp`/faster-whisper 서비스 또는 API 백엔드

## 6. 자막을 넘어선 콘텐츠 확장

- **챕터**: video info/player response의 구조화 데이터 우선. 광고 스트립은 **후처리 정책으로 분리**(획득 로직과 섞지 않음).
- **설명(description)**: youtubei.js video info 또는 공식 API. 줄바꿈/URL 보존.
- **댓글**: youtubei.js `getComments`, `maxComments`로 페이지네이션 + continuation 상태 반환. rate limit/프라이버시 위험 높음 → **opt-in**.
- **썸네일**: 결정적 URL이지만 가용 variant 검증. 파일 다운로드 대신 URL+해상도 반환.
- **오디오 STT**: 별도 capability. `오디오 해석 → 크기/길이 제한 다운로드 → 전사 → temp 삭제 → 캐시`. 최대 길이·바이트 강제, temp 정리, `source: "stt"` 노출.

## 7. 단계별 우선순위 로드맵

1. **즉시 안정화(현 코드 유지 + 품질화)**: stdout 로깅 제거(→stderr), 타입드 에러, zod 입력 검증, structured output, **현재 동작에 fixture 테스트**부터 두른다.
2. **코어 교체**: 수작업 스크래핑/protobuf/클라이언트 위장 → **provider 추상화 + youtubei.js primary**. 캐시·타임아웃·백오프·실패 텔레메트리 추가.
3. **poToken provider 인터페이스**: `PoTokenProvider` 추상화, BgUtils는 optional 구현으로(기본 OFF 또는 명시적 opt-in).
4. **yt-dlp opt-in 폴백**: config 뒤에 두고 ToS/법적 문서화, 회피 기본값 번들 금지.
5. **공식 Data API OAuth 모드**: 본인 채널/인가 콘텐츠 자막용.
6. **STT 폴백**: 엄격한 제한 + 명시적 opt-in + temp 정리 + source 라벨.
7. (원격 필요 시) **Streamable HTTP 트랜스포트** 추가.

## 8. 두 AI의 의견 차이 (참고)

| 쟁점 | codex(설계/보수적) | antigravity(라이브 리서치/실용적) |
|------|-------------------|-----------------------------------|
| BgUtils/poToken | mandatory 금지, optional 인터페이스로 "BYO provider" | `youtubei.js + bgutils-js`를 1순위 권장 경로로 적극 채택 |
| 2차 폴백 | timedtext 단순 폴백, youtube-transcript류 회의적 | `@danielxceron/youtube-transcript` 포크는 쓸 만하다고 평가 |
| timedtext 현황 | 캡션 URL 알면 유효한 폴백 | "사실상 차단" — 매우 낮음 |
| 프록시 | 명시적 config, 순수 로테이션은 리스크, 캐싱 우선 | 대규모 운영 시 주거용 프록시 회전 "필수" |
| 법적 리스크 | ToS 표기 + 회피 기본값 금지 수준 | **DMCA 1201조 집단소송 동향** 강하게 경고 |

**종합 판단**: poToken은 antigravity 말대로 점점 불가피하지만, 공개 배포 패키지의 기본값으로 BgUtils/프록시 회피를 켜는 것은 codex의 보수적 입장(법적·유지보수 리스크)을 따라 **opt-in**으로 둔다. timedtext는 antigravity의 비관적 최신 평가를 신뢰해 **주력으로 삼지 않는다**. 법적으로는 회피 기술을 기본 번들하지 않고 문서로 명시(특히 상업적 대량 수집 시 DMCA 1201 검토 권고).
